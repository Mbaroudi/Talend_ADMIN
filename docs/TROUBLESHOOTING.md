# Troubleshooting

First reflexes:

```bash
docker compose ps                      # what is (not) running / healthy
docker compose logs -f <service>       # follow one service
docker compose logs rundeck-bootstrap  # one-shot provisioning result
```

## Stack startup

### `rundeck-bootstrap` fails or exits non-zero

The bootstrap waits for Rundeck to be healthy, then logs in and provisions the
project. Common causes:

- **Wrong credentials** — `RUNDECK_ADMIN_PASSWORD` in `.env` does not match the
  `admin` line in `rundeck/etc/realm.properties`. Fix both, then
  `docker compose up -d --force-recreate rundeck rundeck-bootstrap`.
- **Rundeck still starting** — first boot can take 1–3 minutes (DB migration).
  Re-run the bootstrap once Rundeck is healthy:
  `docker compose up rundeck-bootstrap`. Provisioning is idempotent
  (fixed job UUIDs), re-running is safe.

### Port already in use

Another local service occupies 4440/9000/9001/8081/3000/9090/9093/8088/8089.
Change the host-side port in `docker-compose.yml` (`"<host>:<container>"`) and,
if it is a UI, the matching `*_PUBLIC_URL` in `.env`.

### Nexus shows no login password

Nexus initialises for several minutes on first boot. The generated admin
password is inside the container:

```bash
docker compose exec nexus cat /nexus-data/admin.password
```

## Job execution

### `ERROR: no artifact for '<JOB>' under /artifacts`

The runner looks for, in order: `/artifacts/<JOB>/<JOB>_run.sh`,
`/artifacts/<JOB>/<JOB>.jar`, `/artifacts/<JOB>.jar`.

- Check what actually exists:
  `docker compose exec talend-runner ls -la /artifacts`
- `JOB_NAME` is case-sensitive and must match the folder/jar base name.
- If the build just ran, check the builder logs:
  `docker compose logs talend-builder`.

### Rundeck cannot reach the runner (SSH errors)

- The SSH key pair is generated by the runner on first boot into the
  `ssh_keys` volume. If Rundeck started **before** the runner generated it,
  restart Rundeck: `docker compose restart rundeck`.
- To rotate or repair the key:
  `docker compose down && docker volume rm talend-admin_ssh_keys && docker compose up -d`
- "Authentication failure" with a custom runner: the public key must be in
  `~<TALEND_RUNNER_USER>/.ssh/authorized_keys` on that node.

### Job runs but my JVM options seem ignored

`JVM_OPTS` is applied by the `run-talend-job` launcher: for jar artifacts the
flags are passed directly; for Talend `_run.sh` launchers the hardcoded
`-Xms/-Xmx` line is rewritten. Verify what the JVM actually received:

```bash
# the launcher logs its computed command first:
#   [run-talend-job] job=... context=... jvm_opts='-Xms512M -Xmx4G' args: ...
```

in the Rundeck execution log. If you run artifacts manually (outside Rundeck),
export `TALEND_JVM_OPTS` before calling `run-talend-job`.

### No node matches the filter / "No nodes selected"

The job dispatches by tag (`RUNNER_TAG` option → `tags: <value>` filter).
The tag must exist on at least one node in
`rundeck/etc/talend-resources.yaml`. After editing that file:
`docker compose restart rundeck`.

### My context parameters are not picked up

`CONTEXT_PARAMS` must be `key=value` pairs separated by `;` or newlines.
Malformed entries are skipped with a `WARN` in the execution log. Values may
contain spaces (`label=Hello World;env=prod`), but then pairs **must** be
separated by `;`.

## Builds

### TOS builder returns 503 "studio is still being provisioned"

The first start downloads Talend Open Studio (~830 MB) into the `tos_studio`
volume; until it finishes the build API refuses work. Follow the progress:

```bash
docker compose exec tos-builder cat /opt/tos/prepare.log
curl -s http://localhost:8080/health   # inside the container: studio_ready
```

The download happens **once** — it survives container restarts (volume). If it
failed (network), restarting the container resumes the download
(`docker compose restart tos-builder`).

### TOS build fails: "no talend.project found"

`BUILDER=tos` expects **raw Studio project sources** (the `talend.project`
file at the repo root or one level down). If your repo contains a Maven
project or a Studio export, use `BUILDER=maven` instead.

### TOS build fails during code generation

```bash
docker compose logs tos-builder
```

- "job not found": `JOB_NAME` must match the job name in `process/`
  (case-sensitive, without version suffix).
- Out of memory: raise `TOS_BUILDER_HEAP` in `.env` (e.g. `4096m`).
- Missing libraries: resolved on demand from Maven Central into the Studio
  m2 (look for `[on-demand]` lines in the build output) — the first build
  needs network access; later builds use the cache. A
  `WARN: no maven mapping` line means the jar is not in the Studio's index:
  drop it manually under
  `/opt/tos/studio/configuration/.m2/repository/<g>/<a>/<v>/`.

### Build fails in `talend-builder`

```bash
docker compose logs talend-builder
```

- The repository must be a Maven project (`pom.xml` at the root) producing a
  Talend "Build Job" zip or a runnable jar under `target/` — see
  [`examples/talend-jobs/`](../examples/talend-jobs/README.md).
- Private repositories: use a token-in-URL clone URL
  (`https://<token>@github.com/org/repo.git`) or extend the builder image with
  your credentials helper.

## Webhooks

### Pushes do not trigger anything

1. Delivery received? `docker compose logs webhook-handler`.
2. **401/403** → `WEBHOOK_SECRET` mismatch between `.env` and the provider's
   webhook configuration.
3. **Ignored push** → the handler only reacts to pushes touching Talend files
   (`*.item`, `*.properties`, `routines/`, `contexts/`) on an allowed branch.
4. The handler must be reachable from the provider — for local tests use a
   tunnel (`ngrok http 8088`, `cloudflared tunnel`, ...).

## Resetting

```bash
docker compose down            # stop, keep data
docker compose down -v         # stop AND DELETE all volumes (full reset)
docker compose up -d --build   # rebuild after changing Dockerfiles/scripts
```

> `down -v` erases Rundeck history, PostgreSQL, MinIO and Nexus data and the
> SSH keys. Everything is re-provisioned on the next start.
